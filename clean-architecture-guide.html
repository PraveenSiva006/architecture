The Four Layers

How the Expense Tracker maps to Clean Architecture
Before diving in, internalize this: Clean Architecture is about protecting your core business logic from the outside world. Frameworks change. Databases change. UIs change. But the rule "an expense must have a positive amount and a valid category" — that never changes. That's what the layers protect.

How to read this diagram: An arrow means "knows about / depends on." Infrastructure knows about Adapters. Adapters know about Use Cases. Use Cases know about Entities. But Entities know nothing about anyone. The inner layers are completely isolated.

Layer 1

Entities

Enterprise Business Rules — the heart of your app
Entities are your pure business objects. They hold the data and the rules that are true about your domain regardless of any technology. A Transaction entity knows that an amount must be positive and a category must exist. It does not know about React, localStorage, or any API.
✓ No framework imports ✓ No database calls ✓ No UI concerns ✓ Pure JavaScript/TypeScript ✓ Contains business rules
src/features/transactions/entities/Transaction.js
// ✦ ENTITY LAYER — No imports from outer layers ever.
// This file has zero dependencies. It knows only itself.

export const TRANSACTION_TYPES = Object.freeze({
  INCOME:  'income',
  EXPENSE: 'expense',
});

export class Transaction {
  constructor({ id, amount, category, description, date, type }) {
    // ── Business Rules: validated here, nowhere else ──
    if (!amount || amount <= 0)
      throw new Error('Amount must be a positive number');

    if (!category?.trim())
      throw new Error('Category is required');

    if (!Object.values(TRANSACTION_TYPES).includes(type))
      throw new Error(`Invalid type: ${type}`);

    this.id          = id ?? crypto.randomUUID();
    this.amount      = Number(amount);
    this.category    = category.trim();
    this.description = description ?? '';
    this.date        = date ?? new Date().toISOString();
    this.type        = type;
  }

  // ── Business methods: belong on the entity ──
  isExpense()  { return this.type === TRANSACTION_TYPES.EXPENSE; }
  isIncome()   { return this.type === TRANSACTION_TYPES.INCOME; }

  signedAmount() {
    return this.isExpense() ? -this.amount : this.amount;
  }

  isSameMonth(year, month) {
    const d = new Date(this.date);
    return d.getFullYear() === year && d.getMonth() === month;
  }
}
src/features/budget/entities/Budget.js
// Budget entity — pure rules, no framework knowledge

export class Budget {
  constructor({ id, category, limitAmount, month, year }) {
    if (limitAmount <= 0)
      throw new Error('Budget limit must be positive');

    if (month < 0 || month > 11)
      throw new Error('Month must be 0–11');

    this.id          = id ?? crypto.randomUUID();
    this.category    = category;
    this.limitAmount = limitAmount;
    this.month       = month;
    this.year        = year;
  }

  // ── Core business rule: is this budget over the limit? ──
  isExceeded(spent) {
    return spent > this.limitAmount;
  }

  remainingAmount(spent) {
    return this.limitAmount - spent;
  }

  percentUsed(spent) {
    return Math.min((spent / this.limitAmount) * 100, 100);
  }
}

Layer 2

Use Cases

Application Business Rules — what your app actually does
Use Cases are the actions your app can perform. Each use case is a single, named operation: AddExpense, GetMonthlyReport, CheckBudgetLimit. They orchestrate entities and call the repository interface — but they never know if the repository talks to localStorage, a REST API, or a database. That decision belongs to the outer layers.
✓ Imports Entities only ✓ Calls repository interface (not implementation) ✓ One use case = one operation ✓ No UI, no HTTP, no framework ✓ Fully testable in isolation
src/features/transactions/usecases/AddExpense.js
// Use Case: only imports from the Entities layer
import { Transaction, TRANSACTION_TYPES } from '../entities/Transaction';

/**
 * AddExpense use case.
 * Receives a repository interface — doesn't know or care
 * whether it talks to localStorage, Supabase, or an API.
 */
export class AddExpense {
  constructor(transactionRepo) {
    this.repo = transactionRepo;
  }

  async execute({ amount, category, description, date }) {
    // 1. Create entity — business rules validated inside constructor
    const transaction = new Transaction({
      amount,
      category,
      description,
      date,
      type: TRANSACTION_TYPES.EXPENSE,
    });

    // 2. Persist via repository interface
    await this.repo.save(transaction);

    // 3. Return the saved entity to the caller
    return transaction;
  }
}
src/features/analytics/usecases/GetMonthlyReport.js
export class GetMonthlyReport {
  constructor(transactionRepo) {
    this.repo = transactionRepo;
  }

  async execute({ year, month }) {
    const all = await this.repo.findAll();

    // Filter using entity method — logic stays on the entity
    const monthly = all.filter(t => t.isSameMonth(year, month));

    const totalIncome = monthly
      .filter(t => t.isIncome())
      .reduce((sum, t) => sum + t.amount, 0);

    const totalExpense = monthly
      .filter(t => t.isExpense())
      .reduce((sum, t) => sum + t.amount, 0);

    // Group expenses by category
    const byCategory = monthly
      .filter(t => t.isExpense())
      .reduce((acc, t) => {
        acc[t.category] = (acc[t.category] ?? 0) + t.amount;
        return acc;
      }, {});

    return {
      transactions: monthly,
      totalIncome,
      totalExpense,
      netBalance: totalIncome - totalExpense,
      byCategory,
    };
  }
}
src/features/budget/usecases/CheckBudgetLimit.js
export class CheckBudgetLimit {
  constructor(budgetRepo, transactionRepo) {
    this.budgetRepo = budgetRepo;
    this.txRepo     = transactionRepo;
  }

  async execute({ category, year, month }) {
    const budget = await this.budgetRepo.findByCategory(category);
    if (!budget) return null;

    const all = await this.txRepo.findAll();
    const spent = all
      .filter(t =>
        t.isExpense() &&
        t.category === category &&
        t.isSameMonth(year, month)
      )
      .reduce((sum, t) => sum + t.amount, 0);

    // Delegate the rule to the Budget entity
    return {
      budget,
      spent,
      isExceeded:    budget.isExceeded(spent),
      remaining:     budget.remainingAmount(spent),
      percentUsed:   budget.percentUsed(spent),
    };
  }
}
🔑

Key insight: AddExpense receives transactionRepo as a constructor argument — it never imports a specific storage implementation. This is called Dependency Inversion. The use case defines what it needs (a repo with a save method), and the outer layer decides how to provide it. This means you can test AddExpense with a fake in-memory repo — no localStorage, no database needed.

Layer 3

Interface Adapters

Translators between your core and the outside world
Adapters are translators. They convert data from the format used by use cases and entities into the format needed by the infrastructure (and back). The Repository is the most important adapter — it's the contract that the use case layer defines, and the infrastructure layer fulfills.
✓ Imports Use Cases & Entities ✓ Defines the repository interface ✓ Controllers translate input → use case ✓ Presenters translate output → view model ✓ DTOs convert raw data to entities
src/features/transactions/adapters/ITransactionRepository.js
/**
 * Repository Interface (the "contract")
 *
 * This defines WHAT the use cases need from storage.
 * It does NOT define HOW storage works — that's Infrastructure.
 *
 * Think of this as a job description:
 * "Whoever fills this role must be able to do these things."
 */
export class ITransactionRepository {
  /** Save a new Transaction entity */
  async save(transaction) {
    throw new Error('save() must be implemented');
  }

  /** Return all Transaction entities */
  async findAll() {
    throw new Error('findAll() must be implemented');
  }

  /** Find a single Transaction by id */
  async findById(id) {
    throw new Error('findById() must be implemented');
  }

  /** Delete a Transaction by id */
  async delete(id) {
    throw new Error('delete() must be implemented');
  }
}

// Any class that extends this and implements all methods
// can be used as the repository — localStorage today,
// Supabase tomorrow. Use cases never change.
src/features/transactions/adapters/TransactionController.js
import { AddExpense }  from '../usecases/AddExpense';
import { DeleteTransaction } from '../usecases/DeleteTransaction';

/**
 * Controller: translates raw UI input into use case calls.
 * It knows about use cases, but not about React components.
 */
export class TransactionController {
  constructor(transactionRepo) {
    this.addExpense = new AddExpense(transactionRepo);
    this.deleteTransaction = new DeleteTransaction(transactionRepo);
  }

  // Receives raw form data → validates/transforms → calls use case
  async handleAddExpense(formData) {
    const { amount, category, description, date } = formData;

    // Input coercion lives here — not in the use case
    return this.addExpense.execute({
      amount: parseFloat(amount),
      category: category.trim(),
      description: description?.trim(),
      date: date || new Date().toISOString(),
    });
  }

  async handleDelete(id) {
    return this.deleteTransaction.execute({ id });
  }
}
src/features/transactions/adapters/TransactionPresenter.js
/**
 * Presenter: converts entity data into view-ready format.
 * The UI just renders what the presenter gives it — no logic.
 */
export class TransactionPresenter {

  toViewModel(transaction) {
    return {
      id:          transaction.id,
      displayAmount: `₹${transaction.amount.toLocaleString('en-IN')}`,
      signedDisplay: transaction.isExpense()
        ? `-₹${transaction.amount.toLocaleString('en-IN')}`
        : `+₹${transaction.amount.toLocaleString('en-IN')}`,
      category:     transaction.category,
      description:  transaction.description || 'No description',
      date:         new Date(transaction.date).toLocaleDateString('en-IN'),
      colorClass:   transaction.isExpense() ? 'expense' : 'income',
    };
  }

  toViewModelList(transactions) {
    return transactions.map(t => this.toViewModel(t));
  }
}
🔑

Key insight: The ITransactionRepository interface is defined in the Adapters layer, not the Infrastructure layer. This means the inner layers own the contract and the outer layers must conform to it — not the other way around. This is the Dependency Inversion Principle applied at the architectural level.

Layer 4

Infrastructure

Frameworks, databases, UI — the swappable details
Infrastructure is where real-world details live: React components, localStorage, API calls, routing. This layer knows about everything but is known by nothing inner. The key implication: you can swap your entire database without touching a single use case or entity.
✓ Implements the repository interface ✓ React components live here ✓ localStorage / API calls live here ✓ Wires everything together (composition root) ✓ The most changeable layer
src/features/transactions/infrastructure/LocalStorageTransactionRepo.js
import { ITransactionRepository } from '../adapters/ITransactionRepository';
import { Transaction } from '../entities/Transaction';

const KEY = 'expense_tracker_transactions';

/**
 * Concrete implementation using localStorage.
 * Tomorrow this could be SupabaseTransactionRepo
 * without changing a single use case.
 */
export class LocalStorageTransactionRepo extends ITransactionRepository {

  async save(transaction) {
    const all = await this.findAll();
    all.push(transaction);
    localStorage.setItem(KEY, JSON.stringify(all));
    return transaction;
  }

  async findAll() {
    const raw = localStorage.getItem(KEY);
    if (!raw) return [];

    // Re-hydrate plain objects back into Transaction entities
    return JSON.parse(raw).map(data => new Transaction(data));
  }

  async findById(id) {
    const all = await this.findAll();
    return all.find(t => t.id === id) ?? null;
  }

  async delete(id) {
    const all = await this.findAll();
    const filtered = all.filter(t => t.id !== id);
    localStorage.setItem(KEY, JSON.stringify(filtered));
  }
}
src/features/transactions/infrastructure/useTransactions.js
import { useState, useEffect, useCallback } from 'react';
import { TransactionController } from '../adapters/TransactionController';
import { TransactionPresenter } from '../adapters/TransactionPresenter';
import { LocalStorageTransactionRepo } from './LocalStorageTransactionRepo';

// Wire up the dependencies once (Composition Root pattern)
const repo       = new LocalStorageTransactionRepo();
const controller = new TransactionController(repo);
const presenter  = new TransactionPresenter();

export function useTransactions() {
  const [viewModels, setViewModels] = useState([]);
  const [loading,    setLoading]    = useState(false);
  const [error,     setError]     = useState(null);

  const load = useCallback(async () => {
    const entities = await repo.findAll();
    setViewModels(presenter.toViewModelList(entities));
  }, []);

  useEffect(() => { load(); }, [load]);

  const addExpense = async (formData) => {
    setLoading(true);
    try {
      await controller.handleAddExpense(formData);
      await load();
    } catch (err) {
      setError(err.message); // Business rule errors surface here
    } finally {
      setLoading(false);
    }
  };

  return { viewModels, loading, error, addExpense };
}
src/features/transactions/infrastructure/AddExpenseForm.jsx
import { useState } from 'react';
import { useTransactions } from './useTransactions';

// React component: pure UI. Zero business logic here.
// It just calls addExpense() and renders feedback.
export function AddExpenseForm() {
  const { addExpense, loading, error } = useTransactions();
  const [form, setForm] = useState({
    amount: '', category: '', description: ''
  });

  const handleSubmit = async () => {
    await addExpense(form);
    // Error surfaces from business rules in the entity layer
    if (!error) setForm({ amount: '', category: '', description: '' });
  };

  return (
    <div className="expense-form">
      {error && <p className="error">{error}</p>}
      <input
        type="number"
        placeholder="Amount"
        value={form.amount}
        onChange={e => setForm(f => ({...f, amount: e.target.value}))}
      />
      <input
        type="text"
        placeholder="Category"
        value={form.category}
        onChange={e => setForm(f => ({...f, category: e.target.value}))}
      />
      <button onClick={handleSubmit} disabled={loading}>
        {loading ? 'Adding...' : 'Add Expense'}
      </button>
    </div>
  );
}

 * Composition Root — the ONE place where everything is wired together.
 *
 * This is where you decide: "Today we use localStorage."
 * To switch to Supabase, you change only this file.
 * Every use case, entity, and adapter stays untouched.
 */

  from '../features/transactions/infrastructure/LocalStorageTransactionRepo';
import { LocalStorageBudgetRepo }
  from '../features/budget/infrastructure/LocalStorageBudgetRepo';

  from '../features/transactions/adapters/TransactionController';

  from '../features/budget/usecases/CheckBudgetLimit';

// ── Build the dependency graph ──
const transactionRepo = new LocalStorageTransactionRepo();
const budgetRepo      = new LocalStorageBudgetRepo();

export const container = {
  transactionController: new TransactionController(transactionRepo),
  checkBudgetLimit:      new CheckBudgetLimit(budgetRepo, transactionRepo),

Key insight: Notice that LocalStorageTransactionRepo re-hydrates raw JSON back into Transaction entities. Data from the outside world must always be converted back into entities before use cases touch it. This is what keeps business rules consistent — the entity constructor's validation always runs, no matter where data comes from.

⬡ The Dependency Rule

Source code dependencies
must only point inward.

Nothing in Entities knows about Use Cases, Adapters, or Infrastructure. Nothing in Use Cases knows about React, localStorage, or any framework. Adapters translate between worlds but never leak infrastructure details inward. Only Infrastructure knows everyone — and it's the layer you replace, not the layer you trust.


This is why Clean Architecture survives technology change. The rule "an expense must have a positive amount" will never change. The database you use in 2027 might be completely different. Protect what is permanent. Make what is temporary easy to swap.
